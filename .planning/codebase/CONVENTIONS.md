## TypeScript/React Conventions

### Component Pattern
```typescript
// Functional components with explicit return type
export function ComponentName(): JSX.Element {
  return <div>...</div>;
}

// Or with props interface
interface Props {
  foo: string;
}

export function ComponentName({ foo }: Props): JSX.Element {
  return <div>{foo}</div>;
}
```

### Hook Pattern
```typescript
// use prefix, camelCase
export function useFeatureName() {
  // State, effects, callbacks
  return { state, actions };
}
```

### Import Order (observed)
1. External libraries (React, Next.js)
2. Internal absolute imports (`@/...`)
3. Relative imports (siblings)
4. Types/interfaces

### Type Definitions
```typescript
// Domain types in types.ts
export type PlaybackState = "idle" | "loading" | "playing" | "paused";

export type Voice = {
  name: string;
  path: string;
  size: number;
  is_default: boolean;
};
```

### Error Handling
- Use try/catch for async operations
- Return error states in controller results
- Display via ErrorBanner component

### Styling (Tailwind)
- Utility-first approach
- Common pattern: `className="min-h-screen w-full bg-white"`
- Conditional classes with template literals or clsx

## Python Conventions

### Import Style
```python
# Standard library
import asyncio
from pathlib import Path
from typing import TYPE_CHECKING

# Third party
from fastapi import FastAPI
from pydantic import BaseModel

# Local (absolute within package)
from domain.types import Job
from application.job_service import JobService
```

### Type Hints
```python
# TypedDict for data structures
class Job(TypedDict):
    id: str
    status: str

# Function signatures
def create_job(self, text: str, options: Any) -> str:
    ...

# Optional types
voice: str | None = None
```

### Class Pattern
```python
class JobService:
    def __init__(
        self,
        runtime: "XttsRuntime",
        storage_dir: Path | None = None,
        max_active_jobs: int = 2,
    ):
        self.runtime = runtime
        self.max_active_jobs = max_active_jobs
```

### Async Patterns
- Use `async`/`await` throughout
- Semaphore for concurrency control
- Background tasks via `asyncio.create_task`

## Architecture Conventions

### Layered Structure
Both frontend and backend follow layered architecture:

```
feature/
├── domain/        # Pure business logic
├── application/   # Use cases, orchestration
├── infrastructure/ # External services
└── ui/            # (frontend only) Components
```

### Domain Layer Rules
- No external dependencies
- Pure functions preferred
- Business invariants enforced here

### Application Layer Rules
- Orchestrates domain + infrastructure
- State management
- Hooks (frontend), Services (backend)

### Infrastructure Layer Rules
- External APIs, I/O
- No business logic
- Swappable implementations

## Testing Conventions

## File Organization

### Component Files
- One main component per file
- Co-located sub-components allowed
- Styles via Tailwind (no CSS modules observed)

### Hook Files
- One hook per file
- Named export matching filename

### Domain Files
- `types.ts/py` — All domain types
- Logic split by concern (e.g., `chunkSelection.ts`, `playback.ts`)
