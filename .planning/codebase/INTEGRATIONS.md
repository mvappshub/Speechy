## Backend API (FastAPI)

### Internal Communication
- **Frontend → Backend:** REST API over HTTP
- **Base URL:** `http://localhost:8000` (configurable via `NEXT_PUBLIC_TTS_API_BASE_URL`)
- **CORS:** Full open (`*`) for development

### Key Endpoints

| Endpoint | Method | Purpose |
|----------|--------|---------|
| `/api/health` | GET | Health check + model info |
| `/api/voices` | GET | List available voices |
| `/api/voices` | POST | Upload new voice (WAV) |
| `/api/render` | POST | Start TTS render job |
| `/api/render/{id}` | GET | Get render status |
| `/api/render/{id}/audio` | GET | Get final audio |
| `/api/render/{id}/download` | GET | Download final audio |
| `/api/render/{id}/blocks/{index}/audio` | GET | Get block audio |
| `/api/projects` | GET | List projects |
| `/api/projects` | POST | Create project |
| `/api/projects/sync` | POST | Sync project text/voice |
| `/api/projects/{id}` | GET | Get project |
| `/api/projects/{id}` | PATCH | Update project |
| `/api/projects/{id}` | DELETE | Delete project |
| `/api/projects/{id}/render` | POST | Render project |
| `/api/projects/{id}/download` | GET | Download project audio |
| `/api/projects/{id}/blocks/{index}/audio` | GET | Get project block audio |

## AI/ML Services

### OmniVoice TTS
- **Source:** Local Python runtime (Hugging Face models)
- **Models downloaded from:** Hugging Face Hub
- **Authentication:** Optional `HF_TOKEN` for faster access
- **Models cached locally:** First run downloads models

### ASR Fallback (Conditional)
- **Used when:** No transcript sidecar file provided
- **Requires:** `torchcodec` + FFmpeg
- **Platform issue:** Windows requires "full-shared" FFmpeg build with DLLs
- **Error behavior:** Fails with clear error instead of hanging

## File System Integrations

### Voice Storage
- **Location:** `tts-server/voices/`
- **Format:** WAV files
- **Default voices:** `speaker.wav` through `speaker6.wav`
- **Bundled transcripts:** `.txt` sidecars for default voices

### Project Storage
- **Location:** `tts-server/tmp-jobs/`
- **Structure:** Per-project JSON metadata + audio blocks
- **Format:** Custom `ProjectStore` implementation

### Audio Cache
- Block-level caching by `cache_key` (voice + text hash)
- Final audio concatenation on download

## Integration Patterns

### Progressive Rendering
1. Text split into chunks
2. Each chunk rendered independently
3. Playback starts on first available block
4. Background continues rendering remaining blocks

### Project Sync Flow
1. Frontend calls `/api/projects/sync` with text + voice settings
2. Backend computes blocks and cache keys
3. Existing cached blocks reused
4. New blocks queued for rendering
